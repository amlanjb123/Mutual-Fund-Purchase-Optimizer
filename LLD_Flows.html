Client (Investor) — Full LLD Flow
① Registration & Email Verification
1
Signup Form Submission
Client submits: first_name, last_name, email, phone, password (AES-encrypted), date_of_birth, role="investor"
POST /auth/register
Backend: decrypt_payload(password) → validate strength (8+ chars, uppercase, digit, symbol)
Validates validate_age(date_of_birth) → must be 18+
DB Creates User record + Investor record with date_of_birth
DB Creates OTP record (10-min TTL) → EMAIL OTP sent via Gmail SMTP (background task)
→ 201 "Registration successful. Please verify your email."
Edge case: If email exists but unverified → resends OTP only
Redirected to /verify-otp page
2
Email OTP Verification
Client enters 6-digit OTP received in email
POST /auth/verify-otp → { email, otp }
DB Checks OTP.otp_code == submitted, OTP.used == False, OTP.expires_at > now()
On success: OTP.used = True, User.is_verified = True
→ 200 "Email verified successfully"
→ 400 if OTP invalid or expired
② Two-Factor Login
3
Login (Password Check + OTP Trigger)
POST /auth/login → { email, password (AES-encrypted) }
Backend: decrypt_payload(password) → verify_password() against bcrypt hash
Checks User.is_verified == True
DB Deletes all old OTPs for this user → Creates fresh login OTP (10-min TTL)
→ 200 { status: "otp_required", email }
OTP is stored but NOT emailed yet — user must click "Resend OTP"
→ 401 Invalid credentials | 403 Email not verified
Redirected to /login-otp page
4
Resend OTP → Verify Login OTP → JWT Issued
Client clicks "Resend OTP" → POST /auth/resend-login-otp → EMAIL login OTP emailed
Client submits OTP → POST /auth/verify-login-otp → { email, otp }
DB Validates OTP (unused, not expired) → OTP.used = True
EMAIL Login notification email sent (background)
Backend: create_access_token({ user_id, role }) + create_refresh_token()
→ 200 { access_token, refresh_token, token_type:"bearer", public_id, role:"investor" }
Frontend stores tokens in localStorage → navigates to /client-dashboard
③ Risk Profile Setup
5
Complete Risk Profile (RiskProfileModal)
Client fills RiskProfileModal: Q1, Q2, Q3, Q4 (1–4 scale), annual_income, duration (years), date_of_birth
PUT /investor/profile → InvestorUpdate payload
Backend calls RiskCalculatorService.calculate_risk_profile() → UserRiskScoreEngine:
  • Stated Risk = (avg(Q1-Q4) - 1) / 3
  • Risk Capacity = 0.4×AgeScore + 0.3×IncomeScore + 0.3×DurationScore
  • New User: Final = 0.5×StatedRisk + 0.5×RiskCapacity
  • Old User (has investments): Uses RevealedRisk (PRE+VT+DT) with alpha decay
DB Saves: stated_risk_score, risk_capacity_score, final_risk_score, risk_profile label
Risk profile label: ≤0.33 → "Low Risk" | ≤0.66 → "Moderate" | >0.66 → "High Risk"
→ 200 InvestorResponse with all risk scores
④ Get AI Fund Recommendations
6
Request Fund Recommendations
Client enters: investment_amount (₹), duration_years, risk_profile_override (optional dropdown)
POST /recommendations/

Risk Resolution:
  • If override selected: UserRiskScore = 0.8×override_stated + 0.2×risk_capacity
  • If no override: uses investor.final_risk_score from DB

Fund Universe Query:
DB MutualFund JOIN FundMetrics WHERE is_active=True AND fqs IS NOT NULL AND frs IS NOT NULL

Risk Compatibility Filter: |FRS - filter_score| ≤ 0.30 (±30% band)

Scoring per fund:
  FinalScore = 0.5×FQS − 0.3×|UserRisk−FRS| + 0.2×DurationFit
  DurationFit = max(0, 1 − |UserDuration−FundIdealDuration| / FundIdealDuration)

Basket sizing:
  ≤₹20L → 4 funds | ≤₹1Cr → 5 funds | ≤₹5Cr → 6 funds | >₹5Cr → max(min_needed×2, 8)

Waterfall Allocation: Iterative proportional distribution, max 25% per fund, respects min_investment
Expected Return: FV = Allocation × (1 + CAGR)^duration (uses 1Y/3Y/5Y CAGR by duration)
→ 200 RecommendationResponse { items[], total_allocated, total_expected_return, explanation }
⑤ Invest in Recommended Funds
7
Place Investment Orders
Client clicks "Invest" on a single fund or "Invest All":
Single fund: POST /investor/invest → { scheme_code, amount, ai_explanation }
Bulk: POST /investor/invest-all → { items:[{scheme_code, amount}], ai_explanation }

Backend checks:
  • fund.min_investment — rejects if amount < minimum
  • Fetches FundMetrics.latest_nav → calculates units = amount / nav
DB Creates PortfolioTransaction(status="PENDING")
EMAIL Sends pending order notification to advisor (advisors@mfpo.com) — background
Risk profile recalculated after investment (revealed risk updated)
→ 200 { message, units, nav_at_purchase }
⑥ Portfolio & Order Tracking
8
View Portfolio (Approved Investments Only)
GET /investor/portfolio
DB Queries PortfolioTransaction WHERE investor_id=me AND status="APPROVED"
Aggregates by scheme_code: total investedAmount, units, currentValue
currentValue = units × latest_nav (live from FundMetrics)
returns = currentValue − investedAmount
→ 200 list[PortfolioItemResponse]
9
View Pending Orders & Full Transaction History
GET /investor/pending-orders → Only status="PENDING" transactions (awaiting advisor)
GET /investor/history → ALL transactions (PENDING + APPROVED + REJECTED), sorted by date desc
Both include: fund name, amount, units, nav_at_purchase, transaction_date, approval_date, ai_explanation
Investor gets email notification when advisor approves or rejects an order
⑦ Session Management
10
Token Refresh / Logout / Password Change
Auto-refresh: On 401 response → Axios interceptor: POST /auth/refresh → { refresh_token }
  DB Checks TokenBlacklist → blacklists old token → returns new pair (rotation)

Change Password: POST /auth/change-password → { current_password, new_password } (both AES-encrypted)
  Validates current password → validates new password strength → updates bcrypt hash

Logout: POST /auth/logout → DB blacklists refresh token → frontend clears localStorage
Advisor — Full LLD Flow
① Registration & Login
1
Advisor Registration (ARN Required)
POST /auth/register → { ..., role:"advisor", arn:"XX 00 00 00 000000 0" }
Backend validates ARN format: regex ^[A-Z]{2}\s\d{2}\s\d{2}\s\d{2}\s\d{6}\s\d{1}$
DB Creates User(role="advisor") + Advisor(arn_number) records
EMAIL OTP sent for email verification
→ Same OTP verify flow as investor (step 2 in client flow)
2
Two-Factor Login (same as investor)
POST /auth/login → OTP → POST /auth/verify-login-otp
JWT role payload: role:"advisor" → Frontend redirects to /advisor-dashboard
GUARD All advisor endpoints: _require_advisor(user) checks role in ("advisor", "admin")
② Client Management
3
View All Investor Clients
GET /advisor/clients
DB Queries all User WHERE role="investor" → joins Investor for each
Returns per client: id, name, email, phone, age (computed from DOB), annual_income, duration
Risk data: risk_profile, final_risk_score, stated_risk_score, revealed_risk_score, user_type (New/Old)
Questionnaire: q1, q2, q3, q4
→ 200 { clients: [...] }
4
View Individual Client Profile & Portfolio
Profile: GET /advisor/clients/:client_id/profile
Returns full detail: pan_number, date_of_birth, risk_capacity_score, all Q answers, user_type

Portfolio: GET /advisor/clients/:client_id/portfolio
DB All PortfolioTransaction for this investor (ALL statuses — including PENDING/REJECTED)
Aggregates by scheme_code: investedAmount, currentValue, units, nav, returns%
→ 200 list of holdings
5
Generate Recommendations for a Client
GET /advisor/clients/:client_id/recommendations?amount=100000
Backend verifies investor.final_risk_score IS NOT NULL (must have completed profile)
Internally calls the same get_recommendations() engine — proxying as the client's user
Uses client's investor.duration as investment duration
Returns full RecommendationResponse — same scoring as client would get
→ 400 if client has not completed risk profile survey
③ Order Approval Workflow
6
View All Orders (PENDING First)
GET /advisor/orders
DB All PortfolioTransaction sorted: status DESC (PENDING before APPROVED) then transaction_date DESC
Per order: transaction_id, client_name, client_email, fund_name, scheme_code, amount
Also: duration, risk_profile, ai_explanation, status, transaction_date, rejection_reason
→ 200 list[PendingOrderResponse]
7
Approve or Reject an Order
APPROVE:
POST /advisor/orders/:transaction_id/approve
DB tx.status = "APPROVED"
tx.approved_by_advisor_id = current_user.id
tx.approval_date = utcnow()
EMAIL Order approved email → client (background)
EMAIL Confirmation email → advisor (background)
→ 200 "Order approved successfully"
Transaction now appears in investor's /portfolio
REJECT:
POST /advisor/orders/:transaction_id/reject → { reason }
DB tx.status = "REJECTED"
tx.rejection_reason = payload.reason
tx.approved_by_advisor_id = current_user.id
EMAIL Order rejected email with reason → client (background)
→ 200 "Order rejected successfully"

Guard: tx.status must == "PENDING" — returns 400 if already processed
④ Profile & Session
8
Profile & Session Management
GET /auth/me → Returns id, email, role, first_name, last_name
POST /auth/change-password → AES-decrypted, strength-validated, bcrypt re-hashed
POST /auth/refresh → token rotation via TokenBlacklist
POST /auth/logout → refresh token blacklisted
Shared profile page (SharedProfile.jsx) allows safe public profile sharing
Admin — Full LLD Flow
① Admin Account Setup
1
Admin Registration (Secret Key Required)
POST /auth/register-admin → { first_name, last_name, email, phone, password, admin_secret }
Backend: admin_secret == settings.ADMIN_SECRET (env var) → 403 if mismatch
Password: AES-decrypted → strength validated
DB Creates User(role="admin", is_verified=True) — auto-verified, no OTP needed
→ 200 "Admin user created successfully"
Admin cannot register via the public /auth/register endpoint (blocked by role check)
2
Login (Same 2FA — OTP Still Required)
POST /auth/login → OTP → POST /auth/verify-login-otp
JWT role payload: role:"admin" → Frontend redirects to /admin-dashboard
GUARD All admin endpoints: _require_admin(user) checks user.role == "admin"
Admin can also access advisor endpoints (advisor guard allows "admin" role)
② Dashboard Analytics
3
System Statistics Overview
GET /admin/stats
DB Runs aggregate queries:
  COUNT(User.id) → total_users, total_investors, total_advisors, total_admins
  COUNT(MutualFund.id) → total_funds, active_funds
  MAX(MutualFund.last_synced_at) → last_nav_sync timestamp
→ 200 AdminStatsResponse
③ User Management
4
List All Users with Role-Specific Data
GET /admin/users
DB All User records ordered by created_at DESC
Base fields: public_id, email, first_name, last_name, role, is_active, is_verified, created_at
Investor extras: pan_number, user_type (New/Old), final_risk_score
Advisor extras: arn_number
→ 200 list[AdminUserResponse]
④ Fund Management (CRUD)
5
List All Funds (Including Inactive)
GET /admin/funds
DB All MutualFund + joins FundMetrics + COUNT(NAVHistory) per fund
Returns: scheme_code, name, fund_house, category, expense_ratio, min_investment, is_active
Metrics: fqs, frs, latest_nav, nav_count, last_synced_at
6
Add New Fund (Auto-fetches from mfapi.in)
POST /admin/funds → { scheme_code, expense_ratio?, min_investment?, fund_category?, benchmark_csv? }
Backend: GET https://api.mfapi.in/mf/{scheme_code} (timeout 15s)
Extracts: scheme_name, fund_house, scheme_type, scheme_category, isin_growth
Auto-infers category from name keywords: liquid/overnight → "liquid" | debt/bond → "debt" | else → "equity"
DB Creates MutualFund(is_active=True, last_synced_at=now())
→ 201 AdminFundResponse
→ 409 if scheme_code already exists | 502 if mfapi.in unreachable
7
Update Fund Fields
PUT /admin/funds/:scheme_code → AdminFundUpdate (all optional fields)
Updatable: expense_ratio, min_investment, fund_category, benchmark_csv, is_active
expense_ratio change → records expense_ratio_updated_at = now()
is_active = False → fund removed from recommendation universe
→ 200 updated AdminFundResponse with fresh metrics + NAV count
8
Deactivate Fund (Soft Delete)
DELETE /admin/funds/:scheme_code
DB Sets fund.is_active = False (NOT deleted from DB — historical data preserved)
Fund excluded from recommendations query (WHERE is_active=True)
→ 200 { detail: "Fund deactivated", scheme_name }
⑤ Data Ingestion Triggers
9
Trigger NAV Data Ingestion
POST /admin/ingest/nav
Runs scripts/ingest_mf_data.py via subprocess in a BackgroundTask
Script fetches latest NAV for all active funds from mfapi.in → stores in nav_history
Also runs automatically daily at 01:00 via APScheduler on startup
→ 200 "NAV ingestion started in background"
10
Trigger Fund Metrics Recomputation
POST /admin/ingest/metrics
Runs scripts/compute_metrics.py via subprocess in a BackgroundTask
Computes from NAV history: CAGR (1Y/3Y/5Y), volatility, Sortino, Beta, Alpha, FQS, FRS, max_drawdown
Updates FundMetrics table — these are used by the Recommendation Engine
→ 200 "Metrics computation started in background"
⑥ Order Oversight
11
View All Approved Transactions (Audit Log)
GET /admin/approved-orders
DB All PortfolioTransaction WHERE status="APPROVED" ordered by transaction_date DESC
Per record includes:
  transaction_id, client_name, client_email, fund_name, scheme_code
  amount, units, nav_at_purchase, transaction_date, status
  advisor_name, advisor_email (from approved_by_advisor_id join)
→ 200 list of approved order audit records
